Concepto y Diseño de un Servicio Digital

Caso: Bookr Salud — API de reservas para servicios de salud

Actividad: Definir, modelar y comparar un servicio digital. Duración de exposición: ~15 minutos.

A. Definición del servicio propuesto

Nombre del servicio

Bookr Salud — Plataforma y API para reservar citas médicas (presenciales o virtuales) con control de disponibilidad en tiempo real.

Problema que resuelve

  • Altos tiempos de espera y reprogramaciones por falta de visibilidad del cupo real.
  • Canales de atención fragmentados (teléfono, WhatsApp, ventanilla) que duplican registros.
  • Dificultad para integrar agendas con historias clínicas y facturación.

Usuarios objetivo

  • Pacientes: buscan y reservan cita por especialidad, ubicación o modalidad.
  • Prestadores (clínicas/consultorios): gestionan horarios, cupos y políticas de cancelación.
  • Integradores (apps de terceros): consumen la API para embebeder reservas.

Propuesta de valor: reservas confiables en tiempo real, con recordatorios inteligentes y APIs fáciles de integrar.

B. Aplicación de las características de un servicio

Intangibilidad

El valor se entrega en forma de acceso y experiencia digital (API/UI), no hay transferencia de un bien físico.

Inseparabilidad

La prestación ocurre mientras el usuario interactúa: la confirmación de cupo y la generación de la cita se producen en el momento de uso.

Variabilidad

Depende de factores como la red del usuario o carga del sistema. Se gestiona con SLOs, escalado automático y pruebas A/B del flujo.

Perecibilidad

Los cupos no usados a una hora específica se pierden. Se mitiga con overbooking controlado, lista de espera y recordatorios.

Ausencia de propiedad

El cliente obtiene derecho de uso (cuenta/API key), no propiedad del software. Términos de servicio y SLAs definen el uso.

Fuente conceptual: definiciones generales de servicio (intangibilidad, inseparabilidad, variabilidad, perecibilidad y ausencia de propiedad).

Flujo de interacción (resumen)

  1. El cliente consulta disponibilidad (fecha, médico, modalidad) vía UI o API.
  2. El API Gateway valida autenticación y enruta al Servicio de Reservas.
  3. Reservas consulta/actualiza inventario de cupos y confirma la cita en DB con bloqueo optimista.
  4. Se emite un evento a Cola para disparar Notificaciones (email/SMS/push) y recordatorios.
  5. Panel Admin gestiona agendas, políticas de cancelación y reportes.

D. Análisis de un servicio real

Servicio elegido: Google Calendar API (servicio de integración y gestión de calendarios).

¿Qué tipo de servicio es?

API REST que expone operaciones para crear, listar y sincronizar eventos de calendario, con OAuth 2.0 y webhooks (watch/notifications).

Aplicación de características teóricas

  • Intangible: acceso a endpoints; no hay producto físico.
  • Inseparable: la experiencia se produce en uso (latencia/estado del servidor).
  • Variable: cuotas, límites, red del cliente influyen en la calidad.
  • Perecedero: tokens expiran; notificaciones pierden valor si llegan tarde.
  • Sin propiedad: los desarrolladores solo tienen licencia de uso según términos/SLAs.

Apéndice: Borrador de API

Ejemplos de endpoints (REST):

GET   /v1/availability?doctorId=...&date=YYYY-MM-DD
POST  /v1/reservations { patientId, slotId, modality }
PATCH /v1/reservations/{id} { status: "cancelled" }
GET   /v1/reservations/{id}
POST  /v1/webhooks { url, events: ["reservation.created"] }

Eventos (cola / webhooks):

reservation.created { id, patientId, datetime, channel }
reservation.reminder { id, sendAt }
reservation.cancelled { id, reason }

SLOs sugeridos: latencia P95 < 300 ms (lectura), < 600 ms (reserva); disponibilidad mensual ≥ 99.9%.
